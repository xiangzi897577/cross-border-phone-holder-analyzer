# PROJECT_PLAN.md

## 项目名称

跨境电商手机支架选品分析平台

## 项目定位

这是一个面向跨境电商卖家的选品分析平台，聚焦手机支架轻小件品类。系统通过模拟 Amazon 候选商品数据与 1688 货源数据，计算商品利润率、竞争强度、物流成本、风险等级和推荐评分，并通过可视化看板辅助选品决策。

## 项目目标

在 2026 年 6 月 15 日前完成一个 React + Node.js 的前后端分离项目，用于深圳中厂暑期前端实习简历。

最终项目需要做到：

- 可以本地完整运行
- 有清晰的前后端目录
- 前端页面完整
- 后端接口可用
- 商品数据来自 Node 接口
- JSON 文件实现轻量数据存储
- 有 Dashboard 数据看板
- 有商品列表、详情、分析、候选池
- 有筛选、搜索、排序
- 有图表可视化
- 有 README、截图、项目介绍
- 能写进简历并用于面试讲解

## 后续扩展边界

从第六阶段开始，本项目不再默认严格禁止以下能力：

* 数据库，例如 MySQL、PostgreSQL、SQLite、MongoDB
* ORM 或数据库工具，例如 Prisma
* UI 组件库，例如 Ant Design、Material UI、其他适合项目的组件库
* 第三方库
* 真实 1688 API
* 真实 Amazon API
* 部署上线
* 图片资源处理
* 数据导入模块
* 登录注册和用户系统

但是必须遵守一个核心原则：

**只做当前明确分发给 Codex 的任务。没有明确要求的功能，不要擅自添加。**

例如：

* 如果当天任务是 UI 优化，就只做 UI 优化，不要顺手接数据库。
* 如果当天任务是部署上线，就只做部署适配，不要顺手重构页面。
* 如果当天任务是补商品图片，就只处理图片字段、图片资源和图片展示，不要顺手接真实 API。
* 如果当天任务是数据库改造，才可以设计并接入数据库。
* 如果当天任务是接入 1688 API，才可以处理真实 API 或数据导入适配。
* 如果当天任务没有要求登录注册，就不要主动加入登录、鉴权和用户权限系统。

## 开发原则

1. 每次只完成当前明确任务。
2. 不提前实现后续功能。
3. 不擅自扩大需求范围。
4. 不为了小任务大范围重构项目。
5. 优先保证现有功能稳定。
6. 如果需要引入新依赖，必须说明原因、影响范围和使用方式。
7. 如果任务涉及数据库、部署或真实 API，需要说明改动范围和潜在风险。
8. 修改后必须说明：

   * 修改了哪些文件
   * 如何运行
   * 如何测试
   * 我需要重点理解哪些代码
9. 始终保持项目可以运行。
10. 当前项目以“能展示、能部署、能讲清楚、能写进简历”为最终目标。

# Day 41 - 6月20日：轻量级 RAG 与动态商品知识库

## 今日目标

让 AI 选品助手不只是泛泛聊天，而是能基于当前 Supabase 商品数据进行推荐和分析。

## 核心思路

不做复杂向量数据库，先实现轻量级 RAG：

```txt
用户提问
↓
后端读取 Supabase 当前 products 数据
↓
根据问题和业务规则筛选相关商品
↓
把相关商品作为上下文传给大模型
↓
AI 基于真实商品数据生成推荐
```

这样 products 表就是当前项目的动态知识库。

当商品数据新增、修改、删除后，AI 下一次查询读取到的就是最新数据。

## 任务

### 商品上下文构造

后端新增或整理：

```txt
server/services/productContextService.js
```

职责：

* 查询 Supabase products 表
* 根据推荐评分取 Top 商品
* 根据利润率、销量、评分、竞争指数、物流成本筛选商品
* 控制传给大模型的数据量
* 将商品数据格式化为 AI 可读文本

### AI Prompt 升级

Prompt 中加入商品上下文：

```txt
以下是当前平台中的部分商品数据，请你只能基于这些商品数据进行分析，不要编造不存在的商品。
```

商品上下文示例：

```txt
商品1：
名称：折叠桌面手机支架
类目：桌面支架
利润率：42%
月销量：860
评分：4.6
竞争指数：35
物流成本：低
推荐评分：88
```

### 多轮对话升级

将 `/api/ai/chat` 请求体升级为：

```json
{
  "messages": [
    { "role": "user", "content": "帮我推荐适合新手的产品" },
    { "role": "assistant", "content": "建议优先看折叠桌面支架..." },
    { "role": "user", "content": "那物流成本低的呢？" }
  ]
}
```

前端只保留当前页面会话，不存数据库。

## 验收标准

* AI 能基于真实商品数据回答
* AI 能推荐当前 products 表中存在的商品
* AI 不明显编造不存在的商品
* 商品数据更新后，AI 下一次回答会读取最新数据
* 支持基础多轮追问
* 前端聊天体验稳定

## 注意事项

* 不要声称已经实现完整向量数据库 RAG
* 可以描述为“轻量级 RAG”或“基于 Supabase 实时商品数据的动态知识库”
* 控制传给大模型的数据量，避免 prompt 过长
* 不要把全部商品无脑塞给大模型
* 优先保证推荐结果可解释

# 本阶段注意事项

1. 不要为了追求高级感而破坏项目稳定性。
2. 不要一开始就做复杂 Agent、向量数据库、长期记忆。
3. products 迁移 Supabase 前，先保证代码整理完成。
4. AI 功能必须通过后端调用，不能在前端暴露 API Key。
5. AI 对话第一版可以不存数据库，刷新页面清空即可。
6. 动态知识库第一版可以基于 Supabase 实时查询，不必接向量数据库。
7. README 和截图应放在 AI 功能完成后再做最终版，避免反复修改。
8. UI 优化要先确定风格，不要让 AI 自由发挥。
9. 项目最终目标是“能展示、能讲清楚、能写进简历”。
